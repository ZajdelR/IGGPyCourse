Moduł 4 - Funkcje i typy mapujące#

  • typy mapujące: dict

  • funkcje

  • dokumentacja

Moduł 4 rozszerza pracę nad danymi satelitarnymi o własne funkcje. Zobaczysz, jak wydzielać powtarzające się fragmenty kodu, przekazywać do nich parametry oraz przygotowywać czytelne dokumentacje. Dzięki temu cokolwiek robisz, stanie się powtarzalne i łatwe do utrzymania.

Funkcje#

  • funkcje (w programowaniu) to “wywoływalne bloki instrukcji”

  • mogą (ale nie muszą) przyjmować argumenty

  • mogą (ale nie muszą) zwracać obiekt (przykład ?)

  • raz napisana funkcja może być wykorzystywana wiele razy

  • zwiększają czytelność kodu

  • łatwiejsze debugowanie

*args i **kwargs#

# po *args mogą występować tylko argumenty kluczowe
# mogą (ale nie muszą) mieć wartości domyślne
def funkcja(*args, kwarg1, kwarg2="key"):
    print("*args =", args)
    print("kwarg1 =", kwarg1)
    print("kwarg2 =", kwarg2)
    
funkcja('a', 'b', 'c', 'd', kwarg1="argument kluczowy")

*args = ('a', 'b', 'c', 'd')
kwarg1 = argument kluczowy
kwarg2 = key

funkcja('a', 'b', 'c', 'd','e','f',kwarg1='34',kwarg2='kwarg2')

*args = ('a', 'b', 'c', 'd', 'e', 'f')
kwarg1 = 34
kwarg2 = kwarg2

# średnia z dowolnej liczby argumentów
def srednia(*args):
    if not args: # brak argumentów -> 0
        return 0
    return sum(args) / len(args)

# *args -> krotka argumentów pozycyjnych
# **kwargs -> słownik argumentów kluczowych
def funkcja(*args,**kwargs):
    print(args)
    for key, value in kwargs.items():
        print(key, value, type(value))

Funkcje w praktyce harmonogramu#

Przygotowując plan pobrań z katalogu scen, funkcja może zebrać wszystkie reguły biznesowe w jednym miejscu. To rozdziela logikę decyzyjną od reszty kodu i ułatwia ponowne wykorzystanie podczas kolejnych kampanii obserwacyjnych.

def plan_download(scene: dict, max_cloud: float = 0.3) -> str:
    """Zwraca decyzję o pobraniu sceny w oparciu o status i zachmurzenie."""
    if scene['status'] != 'READY':
        return f"{scene['mission']} wstrzymana: status {scene['status']}"
    if scene['cloud'] > max_cloud:
        return f"{scene['mission']} odrzucona (zachmurzenie {scene['cloud']:.0%})"
    return f"{scene['mission']} zaakceptowana do pobrania"

scene = {'mission': 'S2A', 'status': 'READY', 'cloud': 0.18}
print(plan_download(scene))

Powyższy przykład zbiera w jednym miejscu warunki if oraz zwraca tekst, który można wykorzystać zarówno w logach, jak i raportach.

Adnotacje typów i wartości domyślne#

Anotowanie parametrów i wartości zwracanych (-> str) ułatwia czytelnikowi zrozumienie kontraktu funkcji, a narzędziom takim jak edytory - podpowiadanie błędów. W danych satelitarnych często operujemy na stringach i liczbach zmiennoprzecinkowych - warto to zaznaczyć w podpisie funkcji.

def format_scene_name(mission: str, date: str, *, level: str = 'L2A') -> str:
    """Buduje nazwę sceny w standardzie MISJA_DATA_POZIOM."""
    return f"{mission}_{date}_{level}"

print(format_scene_name('S2A', '20250301'))
print(format_scene_name('L8', '20240215', level='L1T'))

Struktura docstringa (Args/Returns/Raises)#

Większe funkcje wymagają dokładniejszych opisów: listy parametrów, wartości zwracanej oraz wyjątków. Poniższy przykład korzysta z popularnego formatu kompatybilnego z Sphinx lub NumPy.

def normalize_cloud(cloud: float) -> float:
    """Normalizuje wartość zachmurzenia do zakresu 0-1.

    Parameters
    ----------
    cloud : float
        Zachmurzenie w procentach (0-100) lub ułamku (0-1).

    Returns
    -------
    float
        Zachmurzenie w ułamku.

    Raises
    ------
    ValueError
        Gdy podano liczbę spoza akceptowalnego zakresu.
    """
    if cloud < 0:
        raise ValueError('Zachmurzenie nie może być ujemne.')
    if cloud > 1:
        if cloud <= 100:
            return cloud / 100
        raise ValueError('Wartość przekracza 100%.')
    return cloud

print(normalize_cloud(45))
print(normalize_cloud(0.18))

Docstringi a Sphinx#

Zadbane docstringi z sekcjami Parameters, Returns i Raises są automatycznie wykorzystywane przez narzędzia dokumentacyjne, np. Sphinx. W projektach związanych z analizą danych satelitarnych możesz przygotować katalog funkcji (plan_download, normalize_cloud) i włączyć je do dokumentacji projektu tylko na podstawie komentarzy. Wystarczy odpowiednia struktura docstringa oraz plik konfiguracyjny Sphinx (conf.py), który wskaże moduły do analizy. Dzięki temu użytkownicy Twojego kodu od razu zobaczą opis, listę parametrów i typy zwracanych wartości na wygenerowanych stronach HTML.

Rodzaje błędów i wyjątków#

Moduł 3 pokazał try/except w praktyce. Teraz warto uporządkować najczęściej spotykane błędy:

  • SyntaxError - wykrywany jeszcze przed uruchomieniem programu (np. brak nawiasu).

  • TypeError - wywołanie funkcji z niepoprawnym typem argumentu.

  • ValueError - poprawny typ, lecz zła wartość (np. ujemne zachmurzenie).

  • KeyError - brakujący klucz w słowniku z parametrami sceny.

Znając nazwy wyjątków, możesz reagować bardziej precyzyjnie i zachować informację o źródle problemu.

scene = {'mission': 'S2A', 'cloud': 'brak'}
try:
    cloud = float(scene['cloud'])
    print('Zachmurzenie:', cloud)
except KeyError as err:
    print('Brak pola w rekordzie sceny:', err)
except ValueError as err:
    print('Niepoprawna wartość zachmurzenia:', err)

Grupowanie kilku wyjątków#

Możesz obsłużyć różne wyjątki jednym blokiem lub rozdzielić je dla bardziej szczegółowych komunikatów. W przykładzie poniżej najpierw przechwytujemy ValueError, a następnie ZeroDivisionError.

def compute_normalized_ratio(value: float, scale: float) -> float:
    if scale == 0:
        raise ZeroDivisionError('scale nie może być zerem')
    return value / scale

samples = [(10, 2), ('n/a', 5), (8, 0)]
for value, scale in samples:
    try:
        value = float(value)
        result = compute_normalized_ratio(value, scale)
    except (ValueError, TypeError) as err:
        print(f'Nie można przetworzyć {value!r}:', err)
    except ZeroDivisionError as err:
        print('Skala niepoprawna:', err)
    else:
        print('Wynik:', result)
    finally:
        print('--- koniec iteracji ---')

Dokumentowanie walidacji i wyjątków#

Jeżeli funkcja podnosi wyjątki (raise), wpisz to w docstringu i pokaż przykładowe użycie try/except. Dzięki temu użytkownik wie, jak reagować na problemy z danymi wejściowymi.

def compute_ratio(numerator: float, denominator: float) -> float:
    """Zwraca iloraz dwóch liczb, podnosząc ValueError przy zerowym mianowniku."""
    if denominator == 0:
        raise ValueError('Denominator must be non-zero.')
    return numerator / denominator

try:
    print(compute_ratio(42, 7))
    print(compute_ratio(10, 0))
except ValueError as err:
    print('Błąd podczas obliczeń:', err)
